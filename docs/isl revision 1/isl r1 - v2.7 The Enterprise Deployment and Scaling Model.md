## 1.0 Scope

This document defines the Enterprise Deployment and Scaling Model for the IMHOTEP platform. It establishes how the platform is deployed, scaled, isolated, operated, secured, monitored, recovered, and governed across local, team, enterprise, distributed, hybrid, and cloud-integrated environments.

This document applies to deployments of the IMHOTEP platform itself. It does not define the deployment of systems generated by IMHOTEP, except where platform deployment capabilities affect deployment preparation, artifact repositories, governance, observability, or operational controls.

This document defines:

* deployment profiles
* deployment topology requirements
* local-first deployment behavior
* team and enterprise deployment models
* distributed and cluster deployment models
* hybrid and cloud-integrated deployment models
* multi-project and multi-tenant isolation
* scaling units and scaling policies
* high availability and fault tolerance
* disaster recovery
* enterprise identity and access integration
* enterprise network and security controls
* configuration and deployment automation
* observability and operational monitoring
* governance across deployment environments
* deployment validation and conformance requirements

## 2.0 Purpose of the Enterprise Deployment and Scaling Model

The purpose of the Enterprise Deployment and Scaling Model is to ensure that IMHOTEP can operate reliably across the full range of adoption environments, from a single workstation to an enterprise-scale distributed platform. The architecture must remain consistent across deployment profiles while allowing infrastructure capacity, availability, isolation, and governance controls to increase as organizational use grows.

The IMHOTEP platform performs governed autonomous software construction. In enterprise operation, this requires more than running services. It requires identity integration, project isolation, controlled model and tool execution, durable state, scalable runtime workers, secure artifact storage, telemetry integration, governance enforcement, recovery capability, and operational monitoring.

This model exists to ensure that:

* deployment topology preserves platform architecture
* local-first operation remains supported
* enterprise deployments can scale individual subsystems independently
* multiple projects can run concurrently without cross-contamination
* shared infrastructure does not weaken governance or traceability
* distributed execution preserves state consistency
* runtime workers can scale safely
* model, tool, repository, telemetry, and governance integrations are controlled
* platform operations can recover from component or infrastructure failure
* deployments are reproducible, observable, auditable, and governable

## 4.0 Terms and Definitions

**Deployment Profile**
A named deployment pattern describing the expected scale, topology, operational controls, and infrastructure characteristics of an IMHOTEP platform deployment.

**Deployment Topology**
The arrangement of platform subsystems, infrastructure services, networks, workers, storage systems, and integrations in a deployment.

**Control Plane**
The deployment components responsible for governance, scheduling, configuration, routing, identity, policy, and orchestration control.

**Data Plane**
The deployment components responsible for executing tasks, invoking tools, invoking models, storing artifacts, processing telemetry, and running workload operations.

**Project Isolation**
The separation of specification, artifact, execution, traceability, governance, model context, and telemetry state between projects.

**Tenant Isolation**
The separation of users, teams, organizations, projects, policies, data, and resources in a shared platform deployment.

**Scaling Unit**
A platform component that may be scaled independently.

**Runtime Worker Pool**
A group of workers available to execute runtime work items.

**Deployment Environment**
The infrastructure environment in which the platform runs, such as local workstation, team server, datacenter, Kubernetes cluster, private cloud, public cloud, or hybrid topology.

## 5.0 Deployment Design Principles

### 5.1 Architecture Preservation Across Profiles

Every deployment profile MUST preserve the logical subsystem boundaries defined in ISL v2.0. A local deployment MAY combine subsystems into one process, but it MUST still enforce the logical control boundaries for specification processing, planning, execution, governance, traceability, tool integration, model integration, artifact storage, state, and observability.

### 5.2 Local-First Compatibility

A conforming platform SHOULD support local-first deployment. Local-first deployment means that core platform functions can run without mandatory dependency on external cloud services.

Local-first compatibility MUST NOT prevent enterprise, hybrid, or cloud-integrated deployments.

### 5.3 Independent Subsystem Scaling

Enterprise deployments SHOULD scale subsystems independently. Runtime workers, model inference capacity, tool execution workers, artifact storage, traceability queries, telemetry processing, and governance services may have different capacity requirements.

Scaling one subsystem MUST NOT weaken governance, traceability, validation, audit, or state consistency.

### 5.4 Isolation Before Sharing

A shared enterprise platform MUST isolate projects and tenants before sharing infrastructure resources. Shared infrastructure MAY be used only when isolation boundaries remain enforceable.

### 5.5 Durable State Before Distributed Execution

A deployment MUST NOT enable distributed execution unless durable shared state, recoverable queues, lease coordination, lock coordination, artifact repository consistency, and governance decision consistency are available.

### 5.6 Governance Consistency Across Environments

Governance rules MUST remain enforceable across deployment profiles. A task blocked in an enterprise deployment MUST NOT be allowed merely because it is executed in a local, team, or cloud worker environment.

### 5.7 Observability Is Required for Enterprise Operation

Enterprise deployments MUST provide operational observability sufficient to monitor health, execution progress, failures, governance gates, resource saturation, and recovery behavior.

## 6.0 Deployment Profiles

This section defines standard deployment profiles. Profiles describe scale and operational expectations. They do not prescribe specific vendors or infrastructure products.

### 6.1 Standard Deployment Profiles

| Profile               | Purpose                                                                                              |
| --------------------- | ---------------------------------------------------------------------------------------------------- |
| local-standalone      | Single-user local operation, experimentation, prototyping, and development                           |
| team-shared           | Shared team environment with centralized coordination and repository integration                     |
| enterprise-datacenter | Organization-managed deployment integrated with enterprise infrastructure                            |
| distributed-cluster   | Large-scale deployment with distributed workers and scalable runtime capacity                        |
| hybrid-integrated     | Deployment combining local or enterprise control plane with approved external services               |
| regulated-enterprise  | Deployment for High or Critical risk environments requiring stronger security, audit, and governance |
| multi-tenant-platform | Shared enterprise platform supporting multiple teams, projects, or organizational units              |

### 6.2 Profile Rules

A deployment MUST declare its deployment profile.

A deployment MAY support more than one profile if each profile’s requirements are satisfied.

A deployment profile MUST identify enabled subsystems, storage dependencies, integration dependencies, governance mode, isolation mode, and recovery capability.

A deployment claiming enterprise readiness MUST satisfy at least enterprise-datacenter profile requirements.

## 8.0 Local Standalone Deployment

A local standalone deployment runs on a single workstation or local server. It supports experimentation, specification development, proof-of-concept construction, and local validation.

### 8.1 Local Standalone Requirements

A local standalone deployment SHOULD include:

* Specification Engine
* Semantic Model Engine
* Planning Engine
* Agent Orchestrator
* Model Integration Layer
* Construction Engine
* Tool Integration Layer
* Traceability Engine
* Governance Engine or local governance stub
* Artifact Repository
* State and Memory Store
* Observability emitter or local logs

### 8.2 Local Standalone Rules

A local standalone deployment MAY use embedded or filesystem-backed stores.

A local standalone deployment MAY run all platform subsystems in a single process.

A local standalone deployment MUST still enforce artifact validation, traceability, readiness, and governance rules required by its configured mode.

A local standalone deployment MUST NOT claim enterprise readiness unless it satisfies enterprise profile requirements.

---

## 9.0 Team Shared Deployment

A team shared deployment supports multiple users working against a centralized IMHOTEP instance or coordinated set of services.

### 9.1 Team Shared Requirements

A team shared deployment SHOULD provide:

* centralized specification and canonical model storage
* shared artifact repository integration
* shared construction planning service
* shared execution runtime or worker pool
* identity-aware access control
* team-level governance configuration
* shared telemetry and dashboards
* branch and review workflow support
* state persistence sufficient for recovery

### 9.2 Team Shared Rules

Team deployments MUST identify users and roles.

Team deployments MUST preserve project-level artifact and state separation.

Team deployments SHOULD integrate with shared version control.

Team deployments SHOULD support review workflows before artifact promotion or package generation.

---

## 10.0 Enterprise Datacenter Deployment

An enterprise datacenter deployment runs within organization-managed infrastructure and integrates with enterprise identity, security, observability, governance, repository, and operations systems.

### 10.1 Enterprise Datacenter Requirements

An enterprise datacenter deployment MUST provide:

* durable state stores
* controlled artifact repositories
* identity and access management integration
* governance engine integration
* audit-capable governance records
* traceability storage and query capability
* runtime worker management
* tool execution isolation
* model provider governance
* telemetry collection and alerting
* backup and recovery procedures
* environment-specific configuration management

### 10.2 Enterprise Datacenter Rules

Enterprise deployments MUST separate control-plane and data-plane responsibilities logically.

Enterprise deployments SHOULD support subsystem-level scaling.

Enterprise deployments MUST protect governance, traceability, artifact, and state stores against unauthorized modification.

Enterprise deployments MUST define operational ownership for platform health, governance, security, and recovery.

---

## 11.0 Distributed Cluster Deployment

A distributed cluster deployment supports large-scale task execution across multiple workers, nodes, or compute pools.

### 11.1 Distributed Cluster Requirements

A distributed cluster deployment MUST provide:

* durable execution queues
* distributed worker registration
* worker heartbeat monitoring
* lease coordination
* lock coordination
* distributed state consistency
* shared artifact repository access
* traceability event ordering
* governance decision consistency
* model and tool workload isolation
* recovery after worker or node failure

### 11.2 Distributed Cluster Rules

Distributed execution MUST follow ISL v2.4 distributed runtime requirements.

Cluster workers MUST NOT rely on local-only state for lifecycle decisions.

Cluster deployments MUST detect stale leases and locks.

Cluster deployments MUST detect and halt split-brain execution.

Cluster deployments MUST isolate project workspaces and artifact paths.

---

## 12.0 Hybrid and Cloud-Integrated Deployment

A hybrid deployment combines local or enterprise-controlled platform components with approved external or cloud-hosted services.

### 12.1 Hybrid Integration Patterns

Hybrid deployments MAY use external services for:

* model inference
* artifact storage
* telemetry storage
* distributed compute
* identity federation
* policy evaluation
* source control integration
* package registries
* deployment environment integration

### 12.2 Hybrid Deployment Rules

Hybrid deployments MUST declare which services are external.

Hybrid deployments MUST enforce model and data governance before sending context, artifacts, telemetry, or metadata to external systems.

Hybrid deployments MUST preserve local-first operation where the deployment profile requires it.

Hybrid deployments MUST not externalize restricted data unless governance permits it.

Hybrid deployments MUST maintain traceability and audit continuity across local and external services.

---

## 13.0 Regulated Enterprise Deployment

A regulated enterprise deployment supports systems subject to strong governance, compliance, security, audit, retention, or operational assurance requirements.

### 13.1 Regulated Deployment Requirements

A regulated enterprise deployment MUST provide:

* strong identity and access control
* role-based governance enforcement
* separation-of-duties enforcement
* immutable or integrity-protected audit logs
* enhanced traceability retention
* restricted model and tool profiles
* evidence retention policies
* security monitoring
* deployment authorization controls
* backup and disaster recovery
* configuration change governance
* operational runbooks

### 13.2 Regulated Deployment Rules

High and Critical risk systems MUST use regulated deployment controls unless governance explicitly accepts a different deployment profile.

Regulated deployments MUST retain governance, traceability, artifact, and validation evidence according to applicable retention policy.

Regulated deployments MUST not use experimental models, tools, or plugins without explicit governance approval.

## 14.0 Multi-Project and Multi-Tenant Operation

Enterprise deployments may operate many construction pipelines at once. Multi-project operation separates projects. Multi-tenant operation separates organizational or tenant boundaries.

### 14.1 Isolation Domains

The platform MUST support isolation for:

* specifications
* canonical models
* construction plans
* execution graphs
* task queues
* artifact repositories
* traceability graphs
* governance state
* model context packages
* tool workspaces
* telemetry
* configuration
* secrets
* deployment preparation outputs

### 14.3 Isolation Rules

A task from one project MUST NOT read or write another project’s artifacts unless governed cross-project access is explicitly authorized.

A model context package from one tenant MUST NOT include another tenant’s data unless governance explicitly authorizes cross-tenant context.

Traceability graphs MUST be scoped by project and specification version.

Governance state MUST be isolated by project or tenant unless shared policy inheritance is explicitly configured.

Telemetry dashboards MUST enforce project and tenant access controls.

## 15.0 Shared Infrastructure and Resource Pools

Enterprise deployments may share workers, model providers, tools, queues, telemetry stores, and artifact storage across projects.

### 15.1 Shared Resource Types

| Shared Resource     | Isolation Requirement                                |
| ------------------- | ---------------------------------------------------- |
| runtime worker pool | Task namespace, lease, lock, and workspace isolation |
| model provider      | Context and sensitivity isolation                    |
| tool execution pool | Workspace, filesystem, and network isolation         |
| artifact storage    | Namespace and access isolation                       |
| traceability store  | Graph namespace isolation                            |
| telemetry store     | Scope and access isolation                           |
| governance service  | Policy scope and role isolation                      |
| secret manager      | Project or tenant secret scope                       |

### 15.2 Shared Resource Rules

Shared infrastructure MUST enforce logical isolation at minimum.

High and Critical risk systems SHOULD use stronger isolation such as storage, process, network, or cluster isolation.

Shared workers MUST clean task workspaces between executions.

Shared model providers MUST honor context sensitivity and data-retention profiles.

Shared telemetry stores MUST restrict query access by project, tenant, and role.

## 16.0 Scaling Units

This section defines platform components that may scale independently.

### 16.1 Standard Scaling Units

| Scaling Unit             | Scaling Driver                                 |
| ------------------------ | ---------------------------------------------- |
| Specification Engine     | Number and size of specifications              |
| Semantic Model Engine    | Canonical normalization workload               |
| Planning Engine          | Plan generation and adaptation workload        |
| Runtime Scheduler        | Number of active plans and tasks               |
| Runtime Worker Pool      | Task execution volume                          |
| Agent Orchestrator       | Agent invocation volume                        |
| Model Integration Layer  | Model routing and invocation volume            |
| Model Provider Capacity  | Reasoning workload and context size            |
| Tool Integration Workers | Validation, test, scan, and package workload   |
| Artifact Repository      | Artifact count, package size, retention volume |
| Traceability Engine      | Graph size and query workload                  |
| Governance Engine        | Policy checks and approval volume              |
| Telemetry Pipeline       | Event, metric, trace, and log volume           |
| State Store              | State transition volume and query load         |

### 16.2 Scaling Rules

A scaling unit MUST be scaled only in a way that preserves state consistency, traceability, governance enforcement, and artifact integrity.

Runtime workers MAY scale horizontally when durable queues, leases, and locks are available.

Model provider capacity MAY scale independently when routing and governance controls remain enforceable.

Tool workers MAY scale independently when tool sandboxing and workspace isolation remain enforceable.

Traceability query scaling MUST not corrupt traceability graph updates.

### 17.2 Standard Scale-Out Triggers

Scale-out MAY be triggered by:

* ready queue depth
* delayed queue depth
* worker utilization
* model invocation latency
* tool invocation latency
* validation backlog
* telemetry ingestion backlog
* traceability query latency
* artifact repository operation latency
* governance check backlog

### 17.3 Scaling Policy Rules

Scaling policies MUST be versioned.

Scaling policies for regulated environments MUST be governed.

Scale-in MUST NOT terminate workers holding active leases unless draining or recovery behavior is safe.

Scale-out MUST NOT exceed governance, cost, capacity, or risk-tier limits.

## 19.0 High Availability

High availability ensures that platform services remain available despite component failure.

### 19.1 Availability Tiers

| Tier        | Description                                                                             |
| ----------- | --------------------------------------------------------------------------------------- |
| none        | No explicit availability guarantees                                                     |
| basic       | Process restart and local recovery                                                      |
| operational | Redundant critical services and durable state                                           |
| enterprise  | Multi-instance services, failover, durable queues, replicated storage                   |
| regulated   | Enterprise availability plus enhanced audit, recovery testing, and operational controls |

### 19.2 High Availability Requirements

Enterprise and regulated deployments SHOULD provide high availability for:

* State and Memory Store
* Artifact Repository
* Governance Engine
* Traceability Store
* Execution Queues
* Runtime Scheduler
* Telemetry Pipeline
* Identity Integration
* Configuration Store

### 19.3 High Availability Rules

High availability MUST preserve consistency.

Failover MUST not cause duplicate task execution without lease recovery.

Failover MUST not lose governance, traceability, artifact, validation, or state records.

Governance Engine unavailability MUST block governed actions unless an approved degraded-mode policy exists.

## 21.0 Backup and Restore

Backup and restore provide operational protection for platform state and artifacts.

### 21.1 Required Backup Scope

Enterprise backups MUST include:

* specification records
* canonical models
* construction plans
* execution graphs
* state event logs
* artifact metadata
* artifact content
* validation evidence
* repair records
* traceability graphs and snapshots
* governance records and audit logs
* model/provider configuration
* tool configuration
* telemetry retained as audit-support
* deployment profile records
* platform configuration

### 21.2 Restore Rules

Restore MUST preserve identifiers.

Restore MUST preserve specification version context.

Restore MUST preserve audit immutability or integrity markers.

Restore MUST not resume active execution until recovery validation passes.

Restore into a different environment MUST revalidate external service bindings, secrets, identity mappings, and governance configuration.

## 23.0 Network and Connectivity Model

Enterprise deployments must control connectivity between subsystems and external services.

### 23.1 Network Zones

| Zone                      | Purpose                                                  |
| ------------------------- | -------------------------------------------------------- |
| user-access-zone          | User interfaces and APIs                                 |
| control-plane-zone        | Governance, scheduling, configuration, routing           |
| worker-zone               | Runtime workers, tool workers, agent workers             |
| data-zone                 | State stores, artifact repositories, traceability stores |
| model-zone                | Local or enterprise model services                       |
| observability-zone        | Telemetry collection and dashboards                      |
| external-integration-zone | Controlled external services                             |

### 23.2 Network Rules

Worker zones MUST not access governance or state stores except through approved service interfaces.

External model providers MUST be accessed only through the Model Integration Layer.

External tool services MUST be accessed only through the Tool Integration Layer.

Artifact repositories MUST enforce authenticated and authorized access.

Network paths carrying restricted data SHOULD be encrypted.

## 24.0 Secret and Configuration Management

Secrets and configuration are deployment-critical.

### 24.1 Secret Types

Secrets MAY include:

* model provider credentials
* tool service credentials
* repository credentials
* database credentials
* identity provider credentials
* telemetry exporter credentials
* deployment environment credentials
* signing keys
* encryption keys

### 24.2 Secret Management Rules

Secrets MUST NOT be stored in specifications, agent context packages, telemetry logs, or generated artifacts unless explicitly authorized and protected.

Secrets MUST be scoped by project, tenant, environment, and service identity.

Secret access MUST be auditable.

Secret rotation SHOULD be supported in enterprise deployments.

Secret exposure MUST trigger security telemetry and runtime error handling.

### 24.3 Configuration Management Rules

Deployment configuration MUST be versioned.

Configuration changes affecting governance, runtime, model routing, tool execution, artifact repositories, or state stores MUST be auditable.

Configuration changes during active execution MUST trigger impact evaluation when they may affect execution behavior.

## 26.0 Environment Promotion

Environment promotion moves platform configuration or platform deployments between environments.

### 26.1 Environment Stages

| Stage          | Purpose                                        |
| -------------- | ---------------------------------------------- |
| local          | Local development and testing                  |
| dev            | Shared development                             |
| test           | Integration and validation                     |
| staging        | Production-like validation                     |
| prod           | Production platform operation                  |
| regulated-prod | Production for High or Critical risk workloads |

### 26.2 Promotion Rules

Promotion to production MUST require deployment validation.

Promotion to regulated-prod MUST require governance approval.

Configuration promoted between environments MUST preserve version identity.

Secrets MUST be rebound per environment and MUST NOT be copied blindly.

External integrations MUST be revalidated in the target environment.

## 27.0 Deployment Validation

Deployment validation verifies that the platform deployment is operational and compliant with its declared profile.

### 27.1 Validation Checks

Deployment validation MUST check:

* required subsystems are available
* state stores are reachable and durable
* artifact repositories are reachable
* governance engine is operational
* traceability engine is operational
* telemetry pipeline is operational
* identity integration is functional
* required tools are registered or discoverable
* required model providers are registered or available
* worker pools are healthy
* queues are healthy
* backup configuration exists where required
* security controls are configured
* deployment profile requirements are satisfied

### 27.2 Deployment Validation Record Schema

| Field | Type | Required | Description |
|---|---|---|
| deployment-validation-id | string | REQUIRED | Unique validation identifier |
| deployment-profile-id | string | REQUIRED | Deployment profile validated |
| environment | string | REQUIRED | Environment validated |
| validation-checks | array | REQUIRED | Checks performed |
| validation-outcome | enum | REQUIRED | passed, failed, warning |
| findings | array | CONDITIONAL | Validation findings |
| validated-at | ISO 8601 | REQUIRED | Validation time |
| validated-by | string | REQUIRED | Tool, subsystem, or role validating deployment |
| required-action | string | CONDITIONAL | Required remediation when failed or warning |

### 27.3 Validation Rules

A deployment MUST NOT claim a profile unless deployment validation passes for that profile.

A failed deployment validation MUST block production or regulated use.

Warnings MAY be accepted only when governance permits.

## 28.0 Enterprise Observability and Operations

Enterprise deployments require operational observability beyond local logs.

### 28.1 Enterprise Observability Requirements

* model usage dashboards
* tool validation dashboards
* governance dashboards
* artifact repository dashboards
* traceability integrity dashboards
* alerting for critical failures
* telemetry export to enterprise systems

### 28.2 Operations Rules

Critical alerts MUST identify affected project, subsystem, execution, or tenant when determinable.

Operations dashboards MUST enforce access control.

Telemetry export MUST obey sensitivity and governance rules.

Operational runbooks SHOULD exist for critical alert classes.

## 29.0 Governance Across Deployment Environments

Governance must remain enforceable regardless of where the platform runs.

### 29.1 Governance Deployment Requirements

A deployment MUST provide governance enforcement for:

* readiness progression
* execution start
* restricted model use
* restricted tool use
* artifact promotion
* waiver application
* override application
* deployment authorization
* configuration change
* cross-project access
* regulated workload execution

### 29.2 Governance Environment Rules

Governance profiles MAY differ by environment, but weaker lower-environment governance MUST NOT authorize production activity.

Production and regulated environments MUST use explicit governance profiles.

Governance configuration changes MUST be versioned and audited.

A deployment with unavailable Governance Engine MUST block governed actions unless a preapproved degraded-mode policy exists.

## 30.0 Degraded Mode Operation

Degraded mode allows limited platform operation when some non-critical services are unavailable.

### 30.1 Degraded Mode Categories

| Degraded Mode             | Meaning                                                                  |
| ------------------------- | ------------------------------------------------------------------------ |
| read-only                 | Users may view state but not modify or execute                           |
| planning-only             | Specifications may be parsed and plans generated, but execution disabled |
| execution-paused          | Running execution is paused; no new work dispatched                      |
| telemetry-degraded        | Execution continues, but telemetry pipeline is partially impaired        |
| external-service-disabled | External model/tool/cloud services disabled; local alternatives only     |
| recovery-only             | Platform performs recovery and integrity checks only                     |

### 30.2 Degraded Mode Rules

Degraded mode MUST be explicit.

Degraded mode MUST be recorded in platform state.

Governance-controlled actions MUST NOT proceed in degraded mode unless policy permits.

Telemetry-degraded mode MUST not hide audit, governance, traceability, or artifact records.

Recovery-only mode MUST not dispatch new construction work.

## 31.0 Enterprise Deployment Security

This section defines deployment-level security requirements.

### 31.1 Minimum Security Requirements

Enterprise deployments MUST support:

* authenticated user access
* authenticated service-to-service communication
* authorization by role, project, tenant, and environment
* encryption for sensitive network traffic
* protected storage for governance and audit records
* protected artifact repositories
* secret management
* network segmentation or equivalent controls
* tool execution isolation
* model context protection
* telemetry access control
* security monitoring

### 31.2 Security Rules

High and Critical risk workloads SHOULD run in environments with stronger isolation controls.

External integrations MUST be registered and governed.

Security findings in deployment infrastructure MUST be recorded and routed according to governance policy.

Deployment configuration MUST not contain plaintext secrets.

### 32.1 Audit Support Requirements

* execution admission records

## 33.0 Deployment Lifecycle Management

The platform deployment itself has a lifecycle.

### 33.1 Deployment Lifecycle States

| State       | Meaning                                      |
| ----------- | -------------------------------------------- |
| proposed    | Deployment profile or environment proposed   |
| provisioned | Infrastructure provisioned                   |
| configured  | Platform configuration applied               |
| validated   | Deployment validation passed                 |
| active      | Deployment available for use                 |
| degraded    | Deployment operating with reduced capability |
| maintenance | Deployment temporarily under maintenance     |
| suspended   | Deployment intentionally disabled            |
| failed      | Deployment failed validation or operation    |
| retired     | Deployment decommissioned                    |

### 33.2 Deployment Lifecycle Rules

A deployment MUST NOT enter active state until validation passes.

A failed deployment MUST not accept new execution requests.

A retired deployment MUST preserve required records according to retention policy.

Maintenance and degraded states MUST be visible in telemetry and operations dashboards.

## 35.0 Conformance Requirements

### 35.1 Deployment Profile Conformance

A deployment profile conforms to ISL v2.7 if it:

* declares profile type
* identifies enabled subsystems
* identifies control-plane and data-plane locations
* identifies state, artifact, governance, traceability, and telemetry profiles
* identifies isolation mode
* identifies recovery tier
* records deployment profile metadata
* passes deployment validation for the claimed profile

### 35.2 Enterprise Deployment Conformance

An enterprise deployment conforms to ISL v2.7 if it:

* provides durable state storage
* provides controlled artifact repository storage
* integrates identity and access control
* enforces governance controls
* preserves audit records
* preserves traceability records
* supports runtime worker management
* supports telemetry and alerting
* supports backup and recovery
* protects secrets
* validates deployment configuration
* supports configuration versioning

### 35.3 Distributed Deployment Conformance

A distributed deployment conforms to ISL v2.7 if it:

* provides durable queues
* supports worker registration and heartbeat
* supports leases and locks
* preserves distributed state consistency
* isolates project workspaces
* detects worker failure
* detects split-brain execution
* supports recovery after node failure
* preserves governance and traceability consistency

### 35.4 Multi-Tenant Deployment Conformance

A multi-tenant deployment conforms to ISL v2.7 if it:

* isolates tenant specifications
* isolates tenant artifacts
* isolates tenant state
* isolates tenant governance records
* isolates tenant traceability graphs
* isolates tenant model context packages
* isolates tenant telemetry access
* scopes secrets by tenant
* prevents cross-tenant access unless explicitly governed

### 35.5 Regulated Deployment Conformance

A regulated deployment conforms to ISL v2.7 if it:

* enforces strong identity and access controls
* enforces separation of duties
* protects audit records with immutability or integrity controls
* uses restricted model and tool profiles where required
* retains evidence according to policy
* supports disaster recovery testing
* supports deployment validation records
* blocks workloads when required compliance controls are missing

### 35.6 Platform Conformance

A platform conforms to ISL v2.7 if it:

* supports at least one declared deployment profile
* preserves logical platform architecture across deployment profiles
* supports local-first deployment where configured
* supports independent scaling of scaling units where claimed
* supports project isolation in shared deployments
* supports governance consistency across environments
* supports deployment validation
* supports deployment telemetry and operational monitoring
* supports backup, restore, and recovery appropriate to declared profile
* prevents deployment profile claims that have not passed validation

## 36.0 Summary

ISL v2.7 defines the Enterprise Deployment and Scaling Model for the IMHOTEP platform. It establishes how the platform is deployed, scaled, isolated, secured, governed, observed, automated, validated, backed up, restored, and operated across local, team, enterprise, distributed, hybrid, regulated, and multi-tenant environments.

This revision strengthens the original deployment model by turning environment descriptions into operational contracts. It defines deployment profiles, deployment profile records, topology requirements, project and tenant isolation, scaling units, scaling policies, capacity planning, high availability, disaster recovery, identity integration, network zones, secret management, deployment automation, environment promotion, deployment validation, degraded mode, enterprise security, compliance support, lifecycle states, error classes, and conformance requirements.

A conforming IMHOTEP deployment MUST preserve the same architectural, governance, traceability, validation, observability, and state integrity rules regardless of scale. The platform may grow from a local workshop to an enterprise construction infrastructure, but it MUST remain governed, isolated, recoverable, observable, and specification-driven.
