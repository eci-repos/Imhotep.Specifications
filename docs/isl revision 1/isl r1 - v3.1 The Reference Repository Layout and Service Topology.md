## 1.0 Scope

This document defines the reference repository layout and service topology for the IMHOTEP reference implementation. It specifies how source code, platform libraries, deployable services, tests, examples, experiments, documentation, build automation, infrastructure configuration, generated artifacts, and operational assets SHALL be organized within a maintainable implementation repository.

This document applies to the reference implementation repository for the IMHOTEP platform. It does not define the artifact repository layout for systems generated by IMHOTEP; that concern is defined in ISL v2.3. This document defines the repository that contains the platform implementation itself.

This document defines:

* root repository structure
* source module organization
* deployable service organization
* gateway service boundaries
* test project organization
* documentation layout
* example specification layout
* experiment isolation
* infrastructure and deployment assets
* build and automation structure
* package and dependency boundaries
* service topology
* local-first topology
* enterprise topology
* repository evolution rules
* conformance requirements

## 2.0 Purpose of the Reference Repository Layout and Service Topology

The purpose of the Reference Repository Layout and Service Topology is to make the IMHOTEP reference implementation understandable, maintainable, testable, and scalable. ISL v3.0 defines the implementation architecture. This document defines how that architecture is arranged in source control and how the resulting components interact as services.

The repository layout is not cosmetic. It is an implementation control mechanism. A disorganized repository would make it easy for contributors to bypass module boundaries, mix domain logic with adapters, couple runtime orchestration directly to model providers, or place governance and traceability logic in unstable locations. The repository structure must therefore reinforce the platform architecture.

This model exists to ensure that:

* contributors can locate platform responsibilities quickly
* core libraries are separated from deployable services
* platform contracts are separated from adapters
* tests mirror implementation modules
* examples can serve as validation assets
* experiments do not destabilize the core platform
* local-first deployment remains easy
* distributed and enterprise deployment remains possible
* service boundaries can evolve without repository restructuring
* source organization reflects the v2.x and v3.0 architecture

## 3.0 Repository Organization Principles

This section defines the principles governing the repository layout. These principles are mandatory because the repository is the long-lived home of the platform implementation.

### 3.1 Architecture-Aligned Layout

The repository MUST align with the logical platform architecture defined in ISL v2.0 and the reference implementation architecture defined in ISL v3.0.

A contributor SHOULD be able to infer the architectural responsibility of a module from its repository location and package name.

### 3.2 Separation of Libraries and Services

Reusable platform logic MUST be placed in source libraries or modules. Deployable processes MUST be placed in service projects.

A service MAY compose multiple libraries, but reusable domain logic MUST NOT live only inside service entry points.

### 3.3 Contracts Before Adapters

Shared contracts, schemas, request records, response records, error records, and event records MUST be separated from provider-specific adapters.

Model provider adapters, tool adapters, repository adapters, identity adapters, and telemetry exporters MUST depend on stable contracts. Stable contracts MUST NOT depend on adapters.

### 3.4 Local-First Without Architectural Collapse

The repository MUST support running the platform locally as a modular monolith, local service set, or containerized local deployment.

Local-first operation MUST NOT collapse logical boundaries. The local host MAY run multiple modules in one process, but it MUST still use the same contracts and control boundaries used by larger deployments.

### 3.5 Test Structure Mirrors Implementation Structure

Tests SHOULD be organized to mirror implementation modules and services. A contributor SHOULD be able to locate unit, contract, integration, and scenario tests for each module or service.

### 3.6 Experimental Isolation

Experimental work MUST be isolated from core implementation modules until it satisfies promotion criteria.

Experiments MAY explore new agents, planning strategies, model adapters, tool integrations, or repository workflows, but they MUST NOT be required by the stable platform unless promoted through a governed process.

### 3.7 Automation Is Part of the Product

Build scripts, deployment manifests, test automation, validation automation, and developer tooling MUST be versioned with the repository.

A reference implementation that cannot be built, tested, and run reproducibly is incomplete.

## 4.0 Root Repository Structure

This section defines the required root-level repository layout. A conforming repository MAY add directories, but it MUST preserve the meaning of the standard directories defined here.

### 4.1 Standard Root Layout

| Path           | Purpose                                                                                                              |
| -------------- | -------------------------------------------------------------------------------------------------------------------- |
| `/docs`        | ISL documents, architecture documentation, contributor guidance, governance guidance                                 |
| `/examples`    | Example ISL specifications and reference construction scenarios                                                      |
| `/assets`      | Static diagrams, images, sample payloads, and non-code assets                                                        |
| `/src`         | Core platform libraries and modules                                                                                  |
| `/services`    | Deployable platform services                                                                                         |
| `/contracts`   | Shared schemas, API contracts, event contracts, and generated contract artifacts                                     |
| `/adapters`    | Provider-specific model, tool, repository, identity, and telemetry adapters                                          |
| `/tests`       | Automated tests                                                                                                      |
| `/experiments` | Isolated experimental modules and prototypes                                                                         |
| `/infra`       | Infrastructure, deployment, container, local orchestration, and environment assets                                   |
| `/build`       | Build scripts, packaging scripts, release automation, and CI configuration                                           |
| `/tools`       | Developer tools, repository utilities, validation scripts, and local support tools                                   |
| `/samples`     | Sample generated outputs, sample artifacts, and demonstration data                                                   |
| `/config`      | Versioned non-secret configuration templates                                                                         |
| `/schemas`     | Machine-readable schemas for ISL, canonical entities, contracts, and records                                         |
| `/artifacts`   | Local-only generated platform artifacts when needed for development; SHOULD NOT contain production generated systems |

### 4.2 Root Layout Rules

The repository MUST contain `/src`, `/services`, `/tests`, `/docs`, and `/build`.

The repository SHOULD contain `/contracts`, `/schemas`, `/examples`, `/infra`, and `/config`.

Secrets MUST NOT be stored in the repository.

Generated output under `/artifacts` MUST be local, transient, or clearly classified. Production generated systems MUST be stored in artifact repositories governed by ISL v2.3, not in the platform implementation repository.

A source module MUST NOT depend on a deployable service project.

Core domain modules MUST NOT depend on provider-specific adapters.

A module that owns a state model MUST expose state transition contracts rather than allowing external direct mutation.

### 12.2 Experiment Metadata Schema

| Field | Type | Required | Description |
|---|---|---|
| experiment-id | string | REQUIRED | Unique experiment identifier |
| experiment-name | string | REQUIRED | Human-readable name |
| owner | string | REQUIRED | Responsible contributor, team, or role |
| purpose | string | REQUIRED | What the experiment evaluates |
| affected-subsystems | array | REQUIRED | Subsystems potentially affected |

* ownership is assigned
* purpose and scope are documented
* tests exist
* contracts are defined where applicable
* governance review is completed when risk requires it
* compatibility with current ISL versions is documented
* security implications are reviewed where applicable

## 16.0 Package and Dependency Structure

This section defines package and dependency rules. The objective is to prevent dependency direction from undermining architecture.

### 16.1 Dependency Direction

The preferred dependency direction is:

1. `Imhotep.Common`
2. Contracts and schemas
3. Domain modules
4. Application modules
5. Runtime modules
6. Services
7. Adapters
8. Infrastructure and deployment assets

### 16.2 Prohibited Dependencies

Core modules MUST NOT depend on deployable services.

Core domain modules MUST NOT depend on provider-specific adapters.

Governance, traceability, and state modules MUST NOT depend on runtime worker implementations.

Contracts MUST NOT depend on services.

Adapters MUST NOT be required to run unit tests for core domain modules.

### 16.3 Dependency Rules

Dependency direction MUST be validated by automated checks where practical.

Circular dependencies between source modules MUST NOT be permitted.

A module dependency MUST be justified by an architectural relationship.

Experimental modules MUST NOT be dependencies of stable modules.

## 17.0 Service Topology Overview

This section defines the logical service topology for the reference implementation.

At the center of the topology is the Execution Service. The Execution Service coordinates construction workflows, but it does not own every concern. It delegates model invocation to the Model Gateway, deterministic validation to the Tool Gateway, artifact lifecycle operations to the Artifact Service, traceability operations to the Traceability Service, and governance decisions to the Governance Service.

### 17.1 Logical Service Topology

| Service               | Primary Upstream Callers                             | Primary Downstream Dependencies                                                        |
| --------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Specification Service | CLI, UI, API clients                                 | SemanticModel, Readiness, State, Telemetry                                             |
| Planning Service      | CLI, UI, Execution Service                           | SemanticModel, Traceability, Governance, State                                         |
| Execution Service     | CLI, UI, API clients, Scheduler                      | Runtime, Agents, Models, Tools, Repository, Governance, Traceability, State, Telemetry |
| Model Gateway         | Agent module, Execution Service                      | Model adapters, Governance, State, Telemetry                                           |
| Tool Gateway          | Runtime, Validation tasks                            | Tool adapters, Repository, Governance, State, Telemetry                                |
| Artifact Service      | Execution Service, Tool Gateway, Deployment Preparer | Repository adapters, Traceability, Governance, State                                   |
| Traceability Service  | Planning, Runtime, Artifact, Governance              | Traceability Store, State, Telemetry                                                   |
| Governance Service    | All governed services                                | Governance Store, Audit, State, Telemetry                                              |
| Telemetry Service     | All services                                         | Telemetry Store, Exporters                                                             |
| Configuration Service | All services                                         | Configuration Store, Governance                                                        |

### 17.2 Topology Rules

Services MUST communicate through structured contracts.

Services MUST preserve correlation identifiers.

Services MUST respect ownership boundaries.

Services MUST NOT bypass Governance Service for governed actions.

Services MUST NOT bypass State Module for lifecycle state transitions.

## 18.0 Local-First Service Topology

The local-first topology allows all platform services to run locally.

### 18.1 Local Modes

| Mode                | Description                                                                         |
| ------------------- | ----------------------------------------------------------------------------------- |
| in-process          | All modules run inside one process                                                  |
| local-service-set   | Services run as separate local processes                                            |
| containerized-local | Services and stores run in local containers                                         |
| hybrid-local        | Core platform runs locally while configured model or tool providers may be external |

### 18.2 Local-First Rules

Local-first mode MUST support specification parsing, canonical normalization, planning, execution, artifact repository operations, traceability, governance, and telemetry.

Local-first mode MAY use simplified stores.

Local-first mode MUST preserve service and module boundaries through internal contracts even when deployed in-process.

Local-first mode MUST NOT require external model providers if local model providers are configured and sufficient.

Local-first mode MUST support deterministic tool validation when required tools are available locally.

## 20.0 Worker Topology

Runtime work may be executed by worker services.

### 20.1 Worker Types

| Worker            | Responsibility                                                            |
| ----------------- | ------------------------------------------------------------------------- |
| General Worker    | Executes generic runtime work items                                       |
| Agent Worker      | Executes agent invocation work items                                      |
| Model Worker      | Performs model invocation where gateway uses worker-based execution       |
| Tool Worker       | Executes deterministic tool invocations                                   |
| Repository Worker | Performs branch, merge, package, archive, and repository maintenance work |
| Governance Worker | Processes governance workflow tasks and approval monitoring               |
| Telemetry Worker  | Processes telemetry enrichment, redaction, aggregation, and export        |

### 20.2 Worker Rules

Workers MUST register capabilities.

Workers MUST claim work through leases.

Workers MUST emit heartbeats.

Workers MUST respect project, tenant, and environment isolation.

Workers MUST NOT perform work outside declared capabilities.

## 29.0 Conformance Requirements

### 29.1 Repository Layout Conformance

A repository conforms to ISL v3.1 if it:

* includes required root directories
* separates source modules from services
* separates contracts from adapters
* separates stable implementation from experiments
* contains tests organized by test type or module
* contains documentation for architecture and operation
* contains build automation
* contains configuration templates without secrets
* supports local-first execution
* supports repository validation

### 29.2 Source Module Conformance

A source module conforms to ISL v3.1 if it:

* has a declared architectural responsibility
* exposes structured contracts or module interfaces
* avoids prohibited dependencies
* includes tests or test coverage plan
* emits telemetry where lifecycle-relevant
* preserves state, governance, traceability, or repository rules where applicable

### 29.3 Service Topology Conformance

A service topology conforms to ISL v3.1 if it:

* identifies services and dependencies
* preserves control boundaries
* routes model interactions through Model Gateway
* routes deterministic tool interactions through Tool Gateway
* routes artifact lifecycle operations through Artifact Service
* routes governed actions through Governance Service
* routes traceability writes through Traceability Service
* preserves correlation identifiers
* supports local-first deployment profile
* supports enterprise deployment profile where claimed

### 29.4 Adapter Conformance

An adapter conforms to ISL v3.1 if it:

* resides outside core domain modules
* depends on stable contracts
* declares provider or technology capability
* declares supported contract versions
* exposes structured error results
* includes tests or simulator coverage where practical
* does not expose secrets in logs, telemetry, or test fixtures

### 29.5 Platform Repository Conformance

A platform implementation repository conforms to ISL v3.1 if it:

* implements the repository layout or a documented equivalent
* preserves architecture-aligned organization
* supports service topology manifests
* supports build and test automation
* supports contract and schema validation
* supports module and service evolution
* supports security checks
* supports release versioning
* prevents experiments from becoming hidden production dependencies

## 30.0 Summary

ISL v3.1 defines the Reference Repository Layout and Service Topology for the IMHOTEP platform implementation. It establishes how implementation modules, services, contracts, schemas, adapters, tests, documentation, examples, experiments, infrastructure, configuration, build automation, and service topology manifests are organized.

This revision strengthens the original repository layout by making the structure normative and enterprise-ready. It defines required directories, source modules, service projects, contract boundaries, adapter isolation, test organization, documentation layout, experiment promotion, infrastructure layout, build automation, configuration templates, dependency direction, service topology, worker topology, gateway topology, repository validation, topology manifests, release layout, migration rules, security requirements, and conformance requirements.

A conforming IMHOTEP repository MUST be more than a collection of projects. It MUST be an architectural artifact that teaches contributors how the platform is structured, prevents uncontrolled coupling, supports local-first execution, and remains ready for enterprise deployment and long-term evolution.
